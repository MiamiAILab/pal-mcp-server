# Fallback Policy

This document codifies how zen-mcp-server handles model and provider failures
so that future contributors don't accidentally introduce silent capability
degradation.

## Summary

> **The provider-level error path raises, never substitutes. The
> server-level fallback (context reconstruction only) substitutes within
> the same model class — never falls back from a grounded model to a
> non-grounded model.**

If you're adding fallback logic anywhere in the codebase, your change must
respect both halves of that rule.

## What this prevents

A "silent fallback" — where a request for model X fails and the server
quietly returns a response from a different model that lacks X's defining
capability — is worse than a hard error. The caller proceeds thinking they
got X's output and downstream decisions are made on plausible-looking but
wrong data.

The motivating example: `mcp__zen__chat(model="sonar-deep-research", ...)`
must never silently return content generated by a non-grounded host model
when Perplexity is unreachable. The caller asked for grounded research; if
they don't get it, they need to know.

## Provider layer (`providers/openai_compatible.py`)

The provider-level execution path is in
[`OpenAICompatibleProvider._attempt`](../providers/openai_compatible.py).

```
try:
    return self._run_with_retries(operation=_attempt, ...)
except Exception as exc:
    raise RuntimeError(f"{self.FRIENDLY_NAME} API error ...") from exc
```

On any API failure (network, 5xx, 429, timeout), this raises `RuntimeError`.
It does **not** swap to a different model. The Responses-API path is even
more explicit:

```python
# These models require the /v1/responses endpoint for stateful context.
# If it fails, we should not fall back to chat/completions.
```

If you add a new provider, follow this pattern. If you find yourself
writing `except Exception: return some_other_model.generate(...)` inside a
provider's `generate_content`, stop and re-read this doc.

### Provider-specific postprocessing

Providers may override `_postprocess_response` to:

1. Surface extension fields into `ModelResponse.metadata` for caller
   inspection (e.g. Perplexity citations, grounding signals).
2. Validate provider-specific failure modes by raising. Raising from this
   hook triggers the surrounding retry loop, so a transient backend hiccup
   recovers without surfacing bad data.

The `PerplexityModelProvider` uses this to detect Sonar fail-open responses
(HTTP 200 + zero citations + a known refusal pattern) and raise instead of
returning ungrounded text as though it were grounded research.

## Server layer (`server.py`)

`server.py:1116-1208` contains fallback paths that do substitute models,
but they only fire when:

- A continuation/resumed conversation references a model that's no longer
  available (line 1153: `provider is None`).
- Model-context construction fails entirely (line 1124: `ValueError` from
  `ModelContext.from_arguments`).

Neither of these paths is triggered by API failures during a normal new
tool call. The new-call path validates model availability up front at
`server.py:838-858` and raises `ToolExecutionError` with a clear "Model X
not available" message — no silent substitution.

### Class-aware substitution

`get_preferred_fallback_model` in `providers/registry.py` is the helper
that picks substitute models for the context-reconstruction paths above.
When extending it, preserve class-awareness:

- A grounded model (sonar-*) must not fall back to a non-grounded model.
  If no grounded model is available, raise rather than substitute.
- A vision-capable model must not fall back to a text-only model when the
  request involves images.
- A reasoning-mode model should fall back to another reasoning-mode model
  if possible.

The general principle: a fallback is only safe when the caller's
capability requirements are still satisfied. If the substitute lacks a
capability the caller depends on, hard-fail instead.

## Caller-side detection (defense in depth)

Even with the rules above, callers using grounding-dependent tools
(currently Sonia, the research specialist) should verify the response
metadata before trusting the output:

- `metadata["grounding_active"]` should be `True` for sonar-* responses.
- `metadata["citations_count"]` should be greater than zero.

If either signal is wrong, treat the response as a fallback failure and
re-fire once. If the second attempt also fails grounding, fall back to
native `WebSearch`/`WebFetch` and surface the degradation to the caller.
This caller-side guidance lives in agent personas (e.g.
`~/.claude/agents/sonia.md`).
